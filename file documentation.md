## 1. プロジェクト全体像

このプロジェクトは「RAGを用いたゲームNPCの研究」のための最小実装です。

### 何ができるのか

- 世界観テキストを「NPCの知識」としてベクトルDBに登録する
- ユーザーが質問するとベクトル検索で関連知識を引き出す
- 関連知識+キャラ設定+会話履歴をLLMに渡して、キャラらしい応答を生成する
- 評価用の質問リストを一括で投げて結果をMarkdownで保存する

### 採用している技術スタック

|役割       |技術                                                      |理由                      |
|---------|--------------------------------------------------------|------------------------|
|プログラミング言語|Python 3.11+                                            |機械学習・LLM周辺ライブラリが充実しているため|
|ベクトルDB   |FAISS (Facebook AI Similarity Search)                   |高速・ファイルベース・追加サーバ不要      |
|埋め込みモデル  |sentence-transformers の `intfloat/multilingual-e5-small`|日本語対応・軽量(約120MB)・ローカルで動く|
|LLM      |Google Gemini API (`gemini-2.0-flash-exp`)              |無料枠が大きい・日本語性能が高い        |
|設定ファイル形式 |YAML                                                    |人間が読み書きしやすい             |
|環境変数管理   |python-dotenv                                           |APIキーをコードから分離する標準的な方法   |

### ディレクトリ構成

```
joint-research/
├── main.py                    ← プログラム本体
├── npc_config.yaml            ← NPCのキャラ設定
├── eval_questions.yaml        ← 評価用の質問リスト
├── requirements.txt           ← 依存ライブラリ
├── .env                       ← APIキー(各自で作成、Git管理外)
├── .env.example               ← APIキー記入のテンプレート
├── .gitignore                 ← Gitに無視させるファイル一覧
├── README.md                  ← セットアップ手順・使い方
└── data/
    ├── lore.md                ← NPCに与える世界観テキスト
    ├── vector_store/          ← ベクトル検索インデックスの保存先
    │   ├── .gitkeep           ← 空フォルダをGitに含めるための目印
    │   ├── index.faiss        ← FAISSインデックス(ingestで自動生成)

```

## 3. メインプログラム:`main.py`

プロジェクトの中核となるPythonスクリプト。約420行。**全ての機能がこのファイル1本に集約されています**。

研究室メンバーが読みやすいよう、9つのセクションに分かれていて、上から下に読めば全体の処理が理解できる構成です。

### ファイル全体の構造

```python
"""
RAG搭載NPC 最小実装
===================
このスクリプトは「ゲームNPCに知識をRAG(検索拡張生成)で与える」ための最小構成です。

【処理の流れ】
1. ingest: data/lore.md を細かく分割し、各断片をベクトル化してFAISSに保存
2. chat:   ユーザーの発言をベクトル化 → FAISSから関連する断片を検索
           → NPCの設定 + 関連断片 + 会話履歴 を Gemini に渡して応答生成
3. evaluate: eval_questions.yaml の質問を順に投げて結果をMarkdownに保存
"""

# ========== セクション1: インポートと定数 ==========
# ========== セクション2: 埋め込み(Embedding)関連 ==========
# ========== セクション3: ベクトルストア(FAISS)関連 ==========
# ========== セクション4: テキスト分割(チャンキング)==========
# ========== セクション5: 会話履歴管理 ==========
# ========== セクション6: LLM呼び出し ==========
# ========== セクション7: NPCエージェント(本体)==========
# ========== セクション8: 各コマンド実装 ==========
# ========== セクション9: エントリポイント ==========
```

以下、各セクションを詳細に解説します。

### セクション2:埋め込み(Embedding)関連

#### 役割

テキストを数値ベクトル(384次元の浮動小数点配列)に変換する機能を提供。
このベクトル同士の距離が「意味の近さ」を表すため、ベクトル検索の基盤になります。

#### 主な関数

**`load_embedding_model()`**

`intfloat/multilingual-e5-small` モデルをHugging Faceからダウンロードしてロード。
初回は数十秒かかりますが、2回目以降は `~/.cache/huggingface/` にキャッシュされて即ロードされます。

**`embed_texts(texts, is_query=False)`**

複数のテキストを一括で埋め込みベクトルに変換。

重要ポイント:

- E5系モデルは入力に `"passage: "` または `"query: "` のプレフィックスを付ける仕様です。これを付けないと検索精度が大きく落ちます。`is_query=True` のときは `"query: "`、それ以外は `"passage: "` を内部で自動付与します。
- 出力ベクトルは L2正規化されています。これにより、後の FAISS で「内積=コサイン類似度」として扱えます(セクション3参照)。

#### 内部動作のイメージ

```
"わしは賢者マルクスじゃ"
        ↓
"passage: わしは賢者マルクスじゃ"  ← プレフィックス付与
        ↓
[0.023, -0.157, 0.082, ..., 0.041]  ← 384次元ベクトル(正規化済み)
```

#### 研究で触る可能性

- 別の埋め込みモデルへの差し替え(例:`multilingual-e5-large` で精度向上、または日本語特化モデル)
- プレフィックス戦略の変更(キャラ別プレフィックスなど)

-----

### セクション3:ベクトルストア(FAISS)関連

#### 役割

埋め込みベクトルを保存し、高速に類似検索する機能を提供。

#### 主な関数

**`build_vector_store(chunks)`**

チャンクのリストを受け取り、FAISSインデックスとメタデータを構築。

採用しているのは `IndexFlatIP`(内積による全件検索)です。

- ベクトルがL2正規化されているため、内積=コサイン類似度になる
- 全件比較なので件数が増えると遅くなるが、本研究のような数千〜数万チャンク規模では十分高速
- 近似手法(IVF, HNSW)に比べて精度100%

**`save_vector_store(index, metadatas)`**

FAISSインデックスを `index.faiss` に、メタデータを `meta.pkl` に保存。

**`load_vector_store()`**

保存されたインデックスとメタデータを読み込んで返す。
ファイルが見つからない場合は「先にingestを実行してください」というエラーで終了します。

**`search_vector_store(query_vec, k=4)`**

クエリベクトルに近いチャンクを上位 k 件取得。
返り値はチャンク原文・出典・スコアを含む辞書のリスト。

#### メタデータの構造

```python
{
    "text": "魔法戦争は約300年前に起きた...",  # チャンク原文
    "source": "data/lore.md",                # 出典ファイル
    "chunk_id": 5                            # 何番目のチャンクか
}
```

#### 研究で触る可能性

- インデックス種類の変更(`IndexFlatIP` → `IndexIVFFlat` など、スケール時)
- メタデータの拡張(NPC名、トピックタグ、生成日時など)
- 検索結果のフィルタリング(特定NPCの知識だけ引く、など)

### セクション4:テキスト分割(チャンキング)

#### 役割

長い文章を、ベクトル検索に適したサイズの「チャンク」に分割する。

#### 主な関数

**`chunk_text(text, chunk_size=400, overlap=50)`**

日本語向けに、句点(`。`)を優先的に分割境界として使うシンプルな実装。

処理の流れ:

1. テキストを段落(空行区切り)で分割
1. 各段落を句点で分割
1. 句点単位の文を連結しながら、`chunk_size` 文字を超えたら次のチャンクへ
1. オーバーラップ分を次のチャンク冒頭に含める

#### なぜチャンク分割するのか

- LLMに全文を渡すとコンテキスト長を圧迫する
- 質問と関連する部分だけを渡すほうが、応答の精度が上がる
- ベクトル検索の単位として適切な粒度が必要

#### なぜオーバーラップを設けるのか

チャンクの境目で文脈が途切れる問題を緩和するため。
例:「魔法戦争の発端は王の暗殺だった。これにより…」がチャンク境界で切れると、後半チャンクだけ検索ヒットしても文脈が分からなくなります。オーバーラップによって前後の文脈が両方のチャンクに含まれます。

#### 研究で触る可能性

**ここはRAGデータ品質に直結するため、研究で最も触る場所の一つです。**

- 意味的チャンキング(semantic chunking):意味の切れ目で分割
- 階層的チャンキング:細かいチャンクと大きいチャンクを併用
- LLM自動生成データに特化したチャンク戦略

### セクション5:会話履歴管理

#### 役割

過去の会話を保存・読み込みすることで、NPCが会話の流れを覚えていられるようにする。

#### 主な関数

**`save_message(session_id, role, content)`**

`data/conversations/{session_id}.jsonl` に1行追記。

JSONL形式の利点:

- 1行 = 1メッセージなので追記が高速
- ファイル末尾だけ読めば最新が分かる
- 行単位で削除・編集できる

**`load_recent_messages(session_id, n=10)`**

ファイル末尾から `n` 件のメッセージを読み込んで返す。
ファイルが存在しない場合は空リストを返す(新規セッション扱い)。

#### 保存される内容

```jsonl
{"role": "user", "content": "こんにちは、あなたは誰ですか?"}
{"role": "assistant", "content": "わしは賢者マルクスじゃ。古の図書館を守っておる。"}
{"role": "user", "content": "魔法戦争について教えてください"}
{"role": "assistant", "content": "うむ、わしの知る限りでは…"}
```

#### 研究で触る可能性

**ここも研究の核になり得る場所です。** 現在は「最新N件をそのまま返す」というナイーブ実装ですが、

- 要約圧縮:古い発言を要約して情報を保ちつつ短くする
- 重要発話の抽出:LLMで重要度を判定して残す
- ベクトル検索による関連発話のみ取得:今のトピックに関連する過去発言だけ返す
- 多様性維持のための介入:似た発言が続いたときに別トピックを混入させる

などへの拡張が考えられます。

### セクション6:LLM呼び出し

#### 役割

外部のLLM(Gemini API)に問い合わせて応答を取得する。

#### 主な関数

**`call_llm(system_prompt, messages)`**

`google-generativeai` SDKを使ってGemini APIを呼ぶ。

引数:

- `system_prompt`:NPCの人格・制約・知識ベースを含む指示文
- `messages`:過去の会話履歴(`[{"role": "user", "content": "..."}, ...]` の形式)

返り値:

- LLMが生成した応答文字列

#### APIキーの読み込み

`load_dotenv()` で `.env` を読み、`os.environ["GEMINI_API_KEY"]` から取得。
未設定の場合は分かりやすいエラーメッセージで終了します。

#### 研究で触る可能性

**この関数を差し替えればローカルLLMやファインチューニング済みモデルに移行できます。**

- Ollamaでローカルモデル(Llama, Qwen, Gemmaなど)に切り替え
- OpenAI/Anthropic APIに切り替え
- Hugging Face Transformersで自前モデルを直接呼ぶ
- ファインチューニング済みモデルの呼び出し(将来の発展課題)

抽象化されているので、`call_llm` の中身だけ差し替えれば他のコードはそのまま動きます。

### セクション7:NPCエージェント(本体)

#### 役割

NPC設定の読み込み、RAG検索、プロンプト組み立て、LLM呼び出し、履歴保存を一連の処理として実行する。

#### 主な関数

**`load_npc_config(npc_name="sample_npc")`**

`npc_config.yaml`(または指定された別ファイル)を読んで辞書として返す。

**`build_system_prompt(npc_config, retrieved_chunks)`**

NPC設定 + 検索でヒットしたチャンク + 制約を、構造化されたシステムプロンプトに組み立てる。

組み立てるテンプレートのイメージ:

```
あなたは以下のキャラクターを演じてください。

【キャラクター情報】
名前: 賢者マルクス
年齢: 327
役割: 古の図書館の番人
性格: 穏やかで思慮深い
喋り方: 丁寧語、一人称は「わし」

【背景】
かつて魔法戦争を生き延びた歴史の証人...

【制約】
- 現代的な単語(スマホ、SNS)は使わない
- 知らないことは「わしの知る限りでは…」と前置きする

【参照可能な世界観知識】
1. 魔法戦争は約300年前に勃発し...
2. 古の図書館はアルカニア大陸の中央に位置し...
3. 賢者マルクスは魔法戦争を生き延びた最後の証人...
4. 失われた知識を守ることを使命としている...

上記の知識を踏まえて、キャラクターになりきって応答してください。
```

**`chat_with_npc(session_id, user_message, debug=False)`**

実際の対話処理。ステップごとに動作:

1. ユーザー発言を埋め込みベクトルに変換
1. ベクトルストアから関連する世界観チャンクをTop-K件取得
1. NPC設定 + 検索結果 + 直近の会話履歴 でプロンプトを組み立て
1. Geminiに送って応答を取得
1. ユーザー発言とNPC応答の両方を会話履歴に保存

`debug=True` のときは、検索ヒットしたチャンクの内容とスコアを表示します。

#### 研究で触る可能性

- プロンプトテンプレートの改良(キャラ表現の質向上)
- 検索結果の取捨選択(スコア閾値、リランキングなど)
- 多様性維持のための介入ロジック追加
- 複数NPC対応(NPC同士の対話用)

### セクション8:各コマンド実装

#### 役割

`ingest` / `chat` / `evaluate` の各コマンドの中身を実装。

#### 主な関数

**`cmd_ingest()`**

処理の流れ:

1. `data/lore.md` を読み込み
1. `chunk_text()` でチャンク分割
1. 各チャンクを `embed_texts()` でベクトル化
1. `build_vector_store()` でFAISSインデックス構築
1. `save_vector_store()` で保存

各ステップで進捗を表示:

```
[1/4] 世界観テキストを読み込み中... 3373文字
[2/4] テキストをチャンクに分割中... 10チャンク
[3/4] チャンクをベクトル化してFAISSインデックスを構築中... 10ベクトル登録
[4/4] ベクトルストアを保存中... 保存完了
```

**`cmd_chat(session_id, debug=False)`**

処理の流れ:

1. NPC設定とベクトルストアをロード
1. 入力ループ開始
1. ユーザー入力に応じて分岐:
- `/exit` → ループ終了
- `/reset` → セッションファイル削除
- `/debug` → デバッグモードON/OFF切替
- それ以外 → `chat_with_npc()` を呼び出して応答表示

**`cmd_evaluate()`**

処理の流れ:

1. `eval_questions.yaml` を読み込み
1. 各質問に対して `chat_with_npc()` を実行(セッションは1問ごとに独立)
1. 質問・検索ヒットしたチャンクのソース・LLM応答をMarkdown形式で記録
1. `eval_results/eval_YYYYMMDD_HHMMSS.md` に保存

#### 研究で触る可能性

- 通常はあまり触らない
- 評価スクリプトの拡張(複数LLMの応答比較、多様性指標の自動算出など)

### `data/lore.md`

#### 役割

NPCに与える「世界観の知識」を記述したMarkdownファイル。RAGの知識ベース。

#### 現在の内容

約3,373文字、7章構成のダミー世界観:

- 主要な地名
- 登場人物
- 魔法体系
- 古の図書館
- 失われた知識

#### 編集すると何が起こるか

このファイルを編集したあと、必ず `ingest` を再実行する必要があります:

```bash

```

`ingest` は:

1. `lore.md` を読み込み
1. チャンク分割
1. 各チャンクをベクトル化
1. `data/vector_store/` を上書き保存

#### Markdown形式である意味

- 構造化された文章を書ける(見出し、リスト、強調)

- 人間が読みやすい

- 知識を増やすときは、トピックごとに見出し(`## トピック名`)で区切る
- 一文を長くしすぎない(チャンク境界で切れにくくする)
- 矛盾する記述を混ぜない(NPCの応答が一貫しなくなる)

## 6. 自動生成ファイル群

これらは `main.py` が自動的に作成・更新するファイルです。手動で編集する必要はありません。

### `data/vector_store/index.faiss`

#### 役割

FAISSによるベクトル検索インデックスのバイナリファイル。

#### 内容

`lore.md` から作られた全チャンクの埋め込みベクトル(384次元 × チャンク数)を、検索しやすい形式で保存したもの。

#### サイズの目安

10チャンクの場合:約 15KB(384次元 × 10 × 4バイト + メタデータ)
10,000チャンクの場合:約 15MB

#### 編集ガイド

**手動で編集しないこと。** バイナリ形式なので手で書き換えできません。
`ingest` を実行すれば自動的に再生成されます。

#### Git管理について

`.gitignore` で除外されています。理由:

- バイナリファイルでdiffが取れない
- ファイルサイズが大きい
- 誰が `ingest` してもほぼ同じ結果になるので、共有不要

-----

### `data/vector_store/meta.pkl`

#### 役割

FAISSインデックスに対応するメタデータ(原文・出典・チャンクID)を保存するPickleファイル。

#### なぜ別ファイルなのか

FAISSは「ベクトル」しか保存できません。検索結果を人間が解釈できるようにするには、「ベクトル番号 → 元のチャンク情報」の対応表が必要です。
これを Python の `pickle` 形式で保存しています。

#### 内容のイメージ

```python
[
    {"text": "賢者マルクスは...", "source": "data/lore.md", "chunk_id": 0},
    {"text": "魔法戦争は...", "source": "data/lore.md", "chunk_id": 1},
    ...
]
```

#### 編集ガイド

`index.faiss` と同様、手動で編集しません。
`ingest` で自動再生成されます。

#### Pickleの注意点

- 任意のPythonオブジェクトを保存できる便利な形式だが、信頼できないPickleファイルをロードするとセキュリティリスクがある
- このプロジェクトでは自分が生成したファイルしか読まないので問題なし

-----

### `data/conversations/*.jsonl`

#### 役割

セッションごとの会話履歴ログ。

#### ファイル名のルール

`{session_id}.jsonl`

例:`python main.py chat --session test01` で起動すると `test01.jsonl` が作られる。

#### JSONL形式について

「JSON Lines」の略。1行が1つのJSONオブジェクトになっている形式。

```jsonl
{"role": "user", "content": "こんにちは"}
{"role": "assistant", "content": "わしは賢者マルクスじゃ。"}
{"role": "user", "content": "魔法戦争について教えて"}
{"role": "assistant", "content": "うむ、わしの知る限りでは…"}
```

#### JSONLの利点

- 1行追記でメッセージを保存できる(全体を読み書きしなくて済む)
- 末尾だけ読めば最新が分かる
- 巨大化してもメモリに優しい(行単位でストリーミング処理可能)

#### 編集ガイド

- `/reset` コマンドで削除されるので、続きから話したいセッションは別名で保管
- 手動で編集して特定の発言を削除/修正することも可能(行単位なので安全)

#### Git管理について

`.gitignore` で除外されています。理由:

- 個人の対話履歴を共有する必要がない
- 多数のセッションでファイルが増殖する
- 評価結果は別途 `eval_results/` に残す

-----

### `eval_results/eval_YYYYMMDD_HHMMSS.md`

#### 役割

`python main.py evaluate` 実行時の結果を保存するMarkdownファイル。

#### ファイル名

タイムスタンプ付きで重複しないように生成:

```
eval_results/eval_20260527_143052.md
```

#### 内容のイメージ

```markdown
# 評価結果 (2026-05-27 14:30:52)

## q01: 魔法戦争はいつ起きたのですか?

**検索ヒットしたチャンク:**
- data/lore.md (chunk_id=1, score=0.87)
- data/lore.md (chunk_id=3, score=0.72)

**NPC応答:**
> わしの知る限りでは、魔法戦争は約300年前に勃発したのじゃ...

---

## q02: あなたは誰ですか?
...
```

#### 編集ガイド

- 自動生成されるので手動編集不要
- 評価結果を比較するときは複数の `eval_*.md` を見比べる

#### Git管理について

`.gitignore` で除外されています。
ただし、論文に載せる評価結果は別途 `results/` フォルダなどを作って Git管理してもよいでしょう。

## 9. 処理の流れ(コマンド別)

### `python main.py ingest` 実行時

```
[ユーザーが ingest コマンド実行]
        ↓

        ↓
[main.py セクション8: cmd_ingest() を呼び出し]
        ↓
[1] data/lore.md を読み込み
        ↓
[2] chunk_text() でチャンク分割 (セクション4)
        ↓
[3] load_embedding_model() で埋め込みモデルロード (セクション2)
        ↓
[4] embed_texts() で各チャンクをベクトル化 (セクション2)
        ↓
[5] build_vector_store() でFAISSインデックス構築 (セクション3)
        ↓
[6] save_vector_store() で保存 (セクション3)
        ↓

```

### `python main.py chat --session test01 --debug` 実行時

```
[ユーザーが chat コマンド実行]
        ↓
[main.py セクション9: argparseで chat と引数を認識]
        ↓
[main.py セクション8: cmd_chat(session_id="test01", debug=True) を呼び出し]
        ↓
[1] load_npc_config() でNPC設定読み込み (セクション7)
        ↓
[2] load_vector_store() でベクトルストア読み込み (セクション3)
        ↓
[3] 入力ループ開始
        ↓
[ユーザー入力受付]
        ↓
[特殊コマンド判定 (/exit, /reset, /debug)]
        ↓

        ↓

        ↓

```

## 11. 用語集

### RAG (Retrieval-Augmented Generation)

検索拡張生成。LLMに外部知識を検索して与えることで、最新情報や専門知識を扱えるようにする手法。

### 埋め込み (Embedding)

テキストを数値ベクトルに変換すること。意味が近いテキストはベクトル空間でも近い位置になる。

### ベクトルDB

埋め込みベクトルを保存し、類似度で検索できるデータベース。FAISS、ChromaDB、Qdrant、Pineconeなど。

### FAISS (Facebook AI Similarity Search)

Meta(旧Facebook)が開発した高速ベクトル検索ライブラリ。ローカルファイルで動作する。

### チャンク (Chunk)

長文を検索しやすい単位に分割した断片。

### コサイン類似度

2つのベクトルの「向きの近さ」を測る指標。-1〜1の範囲で、1に近いほど似ている。ベクトルを正規化すれば内積と同じ値になる。

### LLM (Large Language Model)

大規模言語モデル。GPT-4、Gemini、Claude、Llamaなどの総称。

### プロンプト

LLMに与える指示文。

### システムプロンプト

LLMに役割や制約を伝える初期指示。会話全体に影響する。

### ハルシネーション

LLMが事実と異なる情報を自信を持って生成してしまう現象。

### モード崩壊 (Mode Collapse)

生成モデルが多様な出力をやめて、特定パターンに収束してしまう現象。LLM同士の対話を長時間続けると起きやすい。

### Distinct-N

生成テキストの多様性指標。N-gram の重複率から多様性を測る。

### Self-BLEU

生成テキスト同士のBLEUスコア。値が低いほど多様性が高い。

### ファインチューニング (Fine-tuning, FT)

学習済みモデルを特定タスク向けに追加学習させること。LoRA、QLoRAなどの軽量手法がある。

### LoRA (Low-Rank Adaptation)

少ない計算資源でLLMをファインチューニングする手法。元のモデルを凍結し、低ランク行列だけを学習する。

### NPC (Non-Player Character)

ゲームのプレイヤーが操作しないキャラクター。ショップ店員、敵キャラ、村人など。

### ペルソナ一貫性

キャラクターが会話を通じて同じ性格・知識・喋り方を保ち続けること。LLMベースのキャラAIで重要な評価軸。
